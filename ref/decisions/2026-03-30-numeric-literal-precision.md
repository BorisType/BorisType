# 012. Сохранение точности числовых литералов

**Date:** 2026-03-30  
**Status:** Accepted

## Context

BorisScript имеет раздельные типы для чисел: `integer` (64-bit целые) и `real` (с плавающей запятой).
Это значит, что литерал вроде `7603264645902515852` — легитимное 64-битное целое число,
которое отлично помещается в `integer`. Проблема возникает на стороне транспилятора:
JavaScript/TypeScript имеет единый тип `Number` (IEEE 754 double), где 11 бит отведены
под экспоненту и 52 бита под мантиссу, поэтому целые числа за пределами
`Number.MAX_SAFE_INTEGER` (2^53 - 1 = 9007199254740991) теряют точность.

TypeScript Compiler API усугубляет ситуацию: свойство `node.text` у `ts.NumericLiteral`
уже содержит lossy-значение, пропущенное через `Number()`.
Например, для литерала `7603264645902515852` в исходнике, `node.text` возвращает
`"7603264645902516000"` — округлённое до ближайшего представимого double.

Предыдущая реализация в lowering использовала `Number(node.text)` для создания IR-ноды,
что сохраняло эту потерю точности в emitted output.

Помимо больших чисел, целевой язык не поддерживает hex (`0xFF`), octal (`0o77`), binary (`0b1010`)
литералы и numeric separators (`1_000_000`) — все они должны быть преобразованы
в обычные десятичные числа.

## Decision

### Использовать `node.getText(sourceFile)` вместо `node.text`

Метод `getText()` возвращает оригинальный текст из исходного кода без какой-либо интерпретации.
Это позволяет получить точное строковое представление литерала, записанного разработчиком.

### Нормализация через `normalizeNumericLiteral()`

Добавлена функция нормализации, которая конвертирует все форматы в десятичные строки:

| Формат           | Пример                      | Нормализация                                        |
| ---------------- | --------------------------- | --------------------------------------------------- |
| Десятичные целые | `42`, `7603264645902515852` | Как есть (точность в строке)                        |
| Hex              | `0xFF`                      | `BigInt("0xFF")` → `"255"`                          |
| Octal            | `0o77`                      | `BigInt("0o77")` → `"63"`                           |
| Binary           | `0b1010`                    | `BigInt("0b1010")` → `"10"`                         |
| Float / exp      | `3.14`, `1e10`              | `String(Number(text))` (precision-safe для float)   |
| Separators       | `1_000_000`                 | Удаляются через `replace(/_/g, "")` до нормализации |

`BigInt` используется **только как промежуточный шаг** для конвертации hex/octal/binary в десятичное —
он не хранится ни в IR, ни где-либо ещё.

### IRLiteral.value остаётся `number`

IR тип `IRLiteral.value` не изменяется — остаётся `string | number | boolean | null`.
Для чисел за пределами safe range `value` будет lossy, но это допустимо:

- Эмиттер использует **только** поле `raw` (строка) для генерации кода
- Ни один pass не читает `value` числовых литералов
- Когда/если pass-у понадобится работать с числами — тогда и будет принято решение
  о расширении типа `value`

## Consequences

**Плюсы:**

- Числовые литералы любого размера сохраняются с точностью до исходного текста
- Все форматы (hex, octal, binary, separators) корректно конвертируются в десятичные
- Нулевые изменения в IR типах — обратная совместимость
- Минимальный объём изменений (только lowering)

**Минусы:**

- `IRLiteral.value` для больших чисел содержит lossy `Number()` — несоответствие с `raw`
- Если pass захочет оптимизировать числовые выражения, потребуется отдельное решение
  для работы с большими числами

## Alternatives Considered

### 1. Хранить `value` как `bigint` для больших целых

Расширить тип `IRLiteral.value` до `string | number | bigint | boolean | null`,
использовать `BigInt` для целых за пределами safe range.

**Отказ:** Усложняет типы IR без реальной пользы — `value` никем не читается для числовых литералов.
Также потребовались бы проверки `typeof value === "bigint"` во всех местах, работающих с `value`.

### 2. Хранить все числа как строки в `value`

Изменить тип `value` так, чтобы числа всегда хранились как строки.

**Отказ:** Ломает семантику `value` для всех литералов. Потребовало бы изменений
во всех местах, которые создают или проверяют `IRLiteral`. Избыточно для данной проблемы.

### 3. Ручной парсинг всех числовых форматов без BigInt

Реализовать конвертацию hex/octal/binary в десятичное через строковую арифметику.

**Отказ:** Лишняя сложность при наличии `BigInt` в Node.js 22+, который уже умеет
парсить все эти форматы. `BigInt` используется только как вспомогательный инструмент
конвертации, не как формат хранения.
