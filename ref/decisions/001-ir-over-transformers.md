# ADR 001: Компиляция на основе IR вместо AST-трансформеров

**Дата:** 2024-11 (решение принято ранее, задокументировано 2024-12)  
**Статус:** Принято  
**Заменяет:** Подход на основе трансформеров (до v0.2.0)

## Контекст

Ранние версии BorisType (v0.1.x) использовали **AST-трансформеры TypeScript** для прямого преобразования синтаксиса TypeScript в BorisScript. Каждая языковая фича (стрелочные функции, шаблонные литералы, замыкания и т.д.) имела выделенный трансформер, модифицирующий AST на месте.

### Проблемы с трансформерами

1. **Зависимости по порядку**
   - Трансформеры выполнялись последовательно (порядок из `tsconfig.json`)
   - Изменения трансформера N могли нарушить предположения трансформера N+1
   - Сложно рассуждать о финальном выводе

2. **Сложность анализа областей видимости**
   - Каждый трансформер должен был понимать scoping независимо
   - Обнаружение замыканий было хрупким и разбросано по множеству трансформеров
   - Нет единого способа отслеживать captured переменные

3. **Edge Cases**
   - Сложные взаимодействия (напр.: стрелочная функция с шаблонным литералом в деструктурированном параметре)
   - Каждая комбинация требовала специальной обработки
   - Покрытие тестами было сложно гарантировать

4. **Поддерживаемость**
   - 30+ отдельных файлов трансформеров (`ref/transformers/`)
   - Дублированная логика (напр.: переименование идентификаторов, отслеживание scope)
   - Изменения требовали координации нескольких трансформеров

5. **Отладка**
   - Промежуточные состояния AST сложно инспектировать
   - Нет чёткой границы между "фичами TS" и "совместимостью BS"
   - Stack traces указывали на внутренности трансформера, не исходный код

### Почему не просто исправить трансформеры?

- **Babel-подобный подход** (по образцу которого были сделаны трансформеры) работает для JS→JS трансляции
- **TS→BS** фундаментально иное - целевой язык имеет суровые ограничения:
  - Нет замыканий native → требует __env chain
  - Нет let/const → требует анализа hoisting
  - Нет безопасности доступа к свойствам → требует bt.getProperty() wrappers

Эти трансформации **cross-cutting** - они затрагивают несколько уровней AST одновременно. Трансформеры по своей природе **локальны** по scope.

## Решение

Принять **промежуточное представление (IR)** pipeline:

```
TypeScript AST
      ↓
[Scope Analyzer]  ← Единый проход анализа
      ↓
[IR Lowering]     ← TS AST → IR ноды
      ↓
[IR]              ← Упрощённое представление
      ↓
[BT Emitter]      ← IR → BorisScript код
```

### Ключевые изменения

1. **Двухфазная компиляция**
   - **Фаза 1 (Lowering):** TS AST → IR (все фичи TS удалены)
   - **Фаза 2 (Emission):** IR → BorisScript код

2. **Единый анализ Scope**
   - Один проход analyzer перед lowering (`analyzer/scope-analyzer.ts`)
   - Строит полное дерево ScopeInfo
   - Определяет все captured переменные заранее

3. **Явные IR ноды**
   - `IRExpression`, `IRStatement`, `IRFunction` и т.д.
   - Нет TypeScript-специфичных конструкций (уже lowered)
   - Легче валидировать и тестировать

4. **BindingManager**
   - Централизованное отслеживание переменных
   - Генерирует уникальные имена для __env/__desc
   - Обрабатывает hoisting, shadowing, capture

5. **Детерминированный вывод**
   - Одинаковый ввод → одинаковый вывод (нет багов зависимостей от порядка)
   - IR можно инспектировать/логировать для отладки
   - Чёткое разделение ответственности

## Реализация

**Расположение:** пакет `bt-ir/`

**Основные компоненты:**
- `analyzer/scope-analyzer.ts` - Анализ scope
- `lowering/` - Трансформация TS AST → IR
- `ir/` - Определения IR нод
- `emitter/` - Генерация IR → BorisScript
- `pipeline/` - Оркестрация (parse → analyze → lower → emit)

**API:**
```typescript
import { compile, compileFile } from "bt-ir";

const result = compile(sourceCode, { compileMode: "module" });
// result.outputs[0].code — BorisScript
```

## Последствия

### Положительные

✅ **Корректность**
- Единый анализ scope ловит edge cases
- Все трансформации видят согласованное представление scope
- Меньше багов из-за взаимодействия фич

✅ **Поддерживаемость**
- IR проще полного TS AST
- Логика Emitter прямолинейна (IR → string)
- Отладка: инспектировать IR, не промежуточные AST состояния

✅ **Расширяемость**
- Новые фичи BorisScript → добавить IR ноды + поддержку в emitter
- Новые фичи TS → добавить lowering логику
- Чёткие точки расширения

✅ **Производительность**
- Анализ scope — O(n) один проход
- Нет избыточных обходов (трансформеры часто повторно обходили AST)
- Возможна инкрементальная компиляция (будущее)

### Негативные

❌ **Порог вхождения**
- Разработчики должны понимать структуру IR
- Два представления для рассуждений (TS AST + IR)

❌ **Размер кода**
- Больше кода, чем в трансформерах (отдельные lowering + emission)
- Определения типов IR добавляют overhead

❌ **Стоимость миграции**
- Вся логика трансформеров требовала переписывания
- Test suite требовал обновления
- Усилия на документацию (этот ADR, [ir-pipeline.md](../architecture/ir-pipeline.md))

### Trade-offs

**Почему IR, а не прямая генерация TS AST → BS код?**
- Прямая генерация всё равно потребует анализа scope
- TS AST слишком сложное (содержит информацию о типах, JSDoc и т.д.)
- IR упрощает цель - только концепции, нужные BorisScript

**Почему не полная интеграция Babel?**
- Плагины Babel имеют те же проблемы зависимостей от порядка
- Babel целится на JS, не на ограниченное подмножество
- Кастомное IR даёт полный контроль над выводом

## Рассмотренные альтернативы

### 1. Оставить трансформеры, добавить слой координации

**Идея:** Добавить "TransformerOrchestrator" для управления зависимостями

**Отклонено, потому что:**
- Не решает фрагментацию анализа scope
- По-прежнему требует защитного программирования каждого трансформера
- Слой координации добавляет сложность без решения корневой проблемы

### 2. Использовать Babel Preset

**Идея:** Написать Babel preset для вывода BorisScript

**Отклонено, потому что:**
- Babel целится на JS, не BorisScript (разная семантика)
- Всё равно потребуются кастомные плагины для __env, bt.getProperty и т.д.
- API плагинов Babel имеет аналогичную проблему зависимостей от порядка

### 3. Написать интерпретатор BorisScript

**Идея:** Вместо транспиляции интерпретировать TS напрямую в Node.js

**Отклонено, потому что:**
- Цель — деплой на WebSoft HCM (нужны .js файлы)
- Интерпретатор не запустится в окружении BorisScript
- Противоречит цели BorisType (расширить BorisScript, не заменить)

### 4. Многопроходные трансформеры

**Идея:** Запускать трансформеры в несколько проходов (как уровни оптимизации)

**Отклонено, потому что:**
- Не объединяет анализ scope
- Увеличивает сложность (какой проход что делает?)
- По-прежнему страдает от ограничений локальных трансформаций

## Статус

**Реализовано:** v0.2.0 (ноябрь 2024)

**Код трансформеров:** Архивирован в `deprecated/transformers/` (удалено)

## Вынесенные уроки

1. **AST трансформеры работают для JS→JS**, не JS→ограниченное-подмножество
2. **Анализ scope должен быть централизован**, не распределённым по трансформациям
3. **IR упрощает отладку** - промежуточное представление инспектируемо
4. **Стоимость миграции была высокой**, но преимущества поддерживаемости оправдывают её
