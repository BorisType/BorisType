# Аудит результатов ревью кодовой базы bt-ir

**Дата аудита:** 2026-03-14 (на основе предоставленных документов)  
**Объект:** Пакет `bt-ir` (~8000 LOC)  
**Источники:** `bt-ir-final-review-composer.md` (Финальная версия с верификацией), `bt-ir-final-review-claude.md` (Консолидированное мнение)

## 1. Executive Summary

Проведен анализ двух независимых ревью кодовой базы транспилятора/компилятора `bt-ir`. Оба документа датированы одним числом, однако структура `bt-ir-final-review-composer.md` указывает на то, что он является результатом последующей верификации утверждений из `bt-ir-final-review-claude.md`.

**Общий вердикт:** Кодовая база находится в стабильном состоянии после Phase 1-3. Критических runtime-багов, подтвержденных обеими сторонами, не обнаружено (кандидаты на баги в try-finally и hoisting были опровергнуты при проверке кода). Основные риски смещены в область **отсутствия диагностик** (silent failures) и **архитектурного долга**, который проявится при масштабировании.

## 4.Unified Priority Matrix (Объединенный план)

На основе обоих документов составлен итоговый ранжированный список действий.

### Tier S — Критично (Исправить немедленно)

1.  **S1: Silent Failures.** Замена `console.warn` на систему диагностик (`ctx.diagnostics`). Возврат `success: false` при `__unknown__`.
2.  **S2: break/continue в try-finally.** Минимум — ошибка компиляции при обнаружении. Полная поддержка — в roadmap.

### Tier A — Высокий приоритет (Текущий спринт)

1.  **A1: Computed Object Keys.** Сейчас молча отбрасываются. Нужна диагностика или реализация.
2.  **A2: Деструктуризация параметров.** Диагностика "not supported".
3.  **A3: Error Boundary.** Обертка `runPasses` в try-catch для понятных ошибок.
4.  **A4: Precedence.** Удалить dead code или интегрировать в emitter (решено: используется в lowering, нужно документировать ограничение для emitter).

### Tier B — Средний приоритет (Рефакторинг)

1.  **B1: pendingStatements.** Рефакторинг side-channel (при модификации lowering).
2.  **B2: Walker Expressions.** Добавить обход выражений для будущих оптимизаций (DCE).
3.  **B3: Exhaustiveness Checks.** Добавить `const _: never = node` для безопасности типов.
4.  **B4: Pass Ordering.** Валидация зависимостей между passes.

### Tier C — Backlog

1.  **C1: Singleton BindingManager.** Решать при внедрении incremental compilation.
2.  **C6: Source Maps.** Отдельный проект (high effort, high value).
3.  **C5: Устаревшие комментарии.** Очистка (quick win).

## 5. Third-Party Assessment (Оценка адекватности)

Как независимый наблюдатель, анализирующий только текст ревью:

### ✅ Сильные стороны процесса ревью

1.  **Верификация утверждений:** Факт того, что утверждения второго мнения были проверены по исходному коду (L233, L177 и т.д.), говорит о высоком качестве процесса. Это не просто "мнение", а аудит.
2.  **Фокус на пользователе:** Приоритизация Silent Failures (S1) выше архитектурного долга (Singleton) абсолютно верна для продакшн-инструмента. Лучше иметь "грязный" код внутри, чем молча ломать код пользователя.
3.  **Честность относительно ограничений:** Признание, что `try-finally` эмулируется через state machine из-за ограничений целевой платформы (BorisScript), и что это фундаментальное ограничение, а не баг IR.

### ⚠️ Зоны риска (на основе текста)

1.  **Технический долг в Walker:** Пункт B2 (Walker не обходит expressions) является серьезным архитектурным ограничением. Если планируется внедрение оптимизаций (DCE, Constant Folding), это потребует переписывания核心 (core) обходчика. Это оценено как Tier B, но может стать блокиратором для Phase 4.
2.  **Глобальное состояние:** Singleton `BindingManager` (C1) и мутабельный `pendingStatements` (B1) создают скрытые зависимости. Это усложнит тестирование (нужно сбрасывать состояние между тестами) и параллелизацию.

### 💡 Рекомендации третьим лицом

1.  **Консолидация знаний:** Объединить `bt-ir-final-review-composer.md` и `bt-ir-final-review-claude.md` в один файл `ARCHITECTURE_REVIEW.md` в репозитории, чтобы избежать путаницы в будущем (как сейчас между File 1 и File 2).
2.  **Automated Guards:** Рекомендацию B3 (Exhaustiveness checks) стоит внедрить не только в ревью, но и как правило линтера (например, `@typescript-eslint/switch-exhaustiveness-check`), чтобы компилятор сам ругался при добавлении новых типов IR.
3.  **Тестирование "Неподдерживаемого":** Для проблемы S1 (Silent Failures) необходимо добавить интеграционные тесты, которые проверяют, что любой unsupported синтаксис **гарантированно** выбрасывает ошибку, а не генерирует `__unknown__`.
4.  **Документирование ограничений:** Явно зафиксировать в README, что `precedence` обрабатывается на этапе lowering через `IR.grouping()`, а emitter ожидает плоские выражения. Это снимет вопрос A2/С3 навсегда.

## 6. Заключение

Предоставленные материалы демонстрируют зрелый процесс инженерного ревью. Критические ошибки были найдены и классифицированы, ложные тревоги были отсеяны проверкой кода.

**Статус проекта:** Готов к переходу в Phase 4 (Оптимизации и Расширение) **только после** закрытия вопросов Tier S (S1, S2). Игнорирование Silent Failures сделает невозможной надежную эксплуатацию компилятора пользователями. Архитектурные долги (Tier B/C) допустимы на текущем этапе, но должны быть взяты в бэклог.
